# 行情数据链路与 exchange_time 乱序分析

## 背景

在开发 `realtime_parquet`（实时按分钟写 Parquet）时，发现 journal 中 exchange_time 存在乱序现象：按 nano_timestamp（journal 写入时间）排序后，exchange_time 并非严格递增。本文从数据链路全流程分析乱序的根因。

---

## 1. 数据链路全景

```
沪深交易所
  │  多通道并行分发（每个 channel 独立推送一组股票的数据）
  ▼
华泰 Insight 服务端
  │  聚合所有 channel，通过 TCP 推送给客户端
  ▼
Insight SDK（闭源）
  │  单线程接收 TCP 字节流 → 反序列化 protobuf
  │  按 channel 亲和性分派到 5 个回调线程
  ▼
Gateway: OnMarketData() 回调          ← 5 个线程并发执行
  │  深拷贝 protobuf → enqueue 到 ConcurrentQueue
  ▼
ConcurrentQueue（缓冲池）             ← tick / order / trade 各一个
  │  跨 channel 数据在此交叉混合
  ▼
Writer 线程                            ← 每种数据类型一个，串行执行
  │  dequeue → ParseFrom() → write_frame()
  ▼
KungFu Journal（mmap 文件）           ← nano_timestamp 在此产生，严格递增
```

---

## 2. 乱序根因：交易所多 channel 并行 + 回调线程并发入队

### 2.1 交易所层面：数据按 channel 分发

沪深交易所不是用单一数据流发送行情，而是分成多个 **channel（行情频道）**，每个 channel 负责一组股票。从 parquet 数据中可以看到：

| 交易所 | Channel 编号 | 数据来源 |
|--------|-------------|---------|
| 深圳 SZ | 2011, 2012, 2013, 2014, 2015 | `record.channelno()` |
| 上海 SH | 1, 2, 3, 4, 5, 6, 20 | `record.channelno()` |

> 代码证据：`insight_types.h:758` — `channelno = record.channelno()`，该字段来自 Insight SDK 的 protobuf。
>
> 沪深使用完全不同的编号体系（SZ 2011~2015 / SH 1~6,20），说明是各交易所自己定义的频道号，Insight 只是透传。

每只股票固定属于一个 channel（order_0930 中 5256 只股票，0 只跨 channel）。**同一 channel 内数据严格有序，跨 channel 无全局时序保证。**

### 2.2 Gateway 层面：5 线程并发入队

```
                        insight_gateway.cpp:190
client->set_handle_pool_thread_count(5);
```

SDK 用 5 个线程并发调用 `OnMarketData()` 回调。同一 channel 的数据始终由同一线程处理（channel 亲和性分派），不同 channel 由不同线程并发处理。

回调函数做两件事（以 order 为例）：

```cpp
// insight_gateway.cpp:77-80
auto order_data = RawOrderData(data);            // 1. 深拷贝 protobuf
stock_order_writer->save_to_queue(order_data);   // 2. 入队 ConcurrentQueue
```

当线程 A 处理 channel 2011 的数据、线程 B 同时处理 channel 2013 的数据时，两者并发 enqueue 到同一个 order queue，**入队顺序取决于 CPU 调度，不取决于 exchange_time**。

`moodycamel::ConcurrentQueue` 的保序特性：**同一生产者线程入队的数据，出队顺序不变**。所以同 channel（同线程）的数据保序，跨 channel（不同线程）的数据乱序。

## 3. 数据验证

分析 2026-03-04 的 replay 数据（order 通道）：

### 3.1 乱序 100% 发生在不同 channel 之间

| 文件 | 总行数 | 同 Channel 乱序 | 不同 Channel 乱序 |
|------|-------|-----------------|-------------------|
| order_0916 | 70,513 | **0** | 11,908 |
| order_0917 | 80,145 | **0** | 13,523 |
| order_0918 | 87,713 | **0** | 14,795 |

**同 Channel 内 exchange_time 严格有序，乱序全部来自跨 Channel 交叉。**

### 3.2 乱序的典型表现

```
先到: SZ 002506  Channel=2013  exchTime=09:16:00.760  nano=09:15:59.xxx
后到: SZ 000711  Channel=2015  exchTime=09:16:00.720  nano=09:15:59.xxx
```

Channel 2013 已推进到 760ms，Channel 2015 还在 720ms。两个 channel 各自内部有序，但彼此进度不同步。

### 3.3 nano_timestamp 严格递增

| 通道 | 总行数 | nano 回退次数 |
|------|-------|-------------|
| tick | 1,090,885 | **0** |
| order | 4,982,609 | **0** |
| trade | 5,587,863 | **0** |

nano_timestamp 是 writer 线程调用 `write_frame()` 时取的墙上时间，单线程串行写入，所以严格递增。

## 4. 数据写入流程详解

### 4.1 ConcurrentQueue 的作用：缓冲池

```
insight_gateway.h:86
moodycamel::ConcurrentQueue<T1> queue{1000000};  // 预分配 100 万个槽位
```

SDK 回调是"推"模式——数据来了必须接住。queue 的作用是让回调尽快返回（只做拷贝 + 入队），把重活（ParseFrom + write_frame）留给 writer 线程异步处理。这样即使 writer 暂时处理不过来，数据先在 queue 里缓住，不会导致 SDK 内部积压丢数据。

#### 4.1.1 底层数据结构：分块链表

`moodycamel::ConcurrentQueue` 不是环形数组，是**分块链表**（block-based linked list）：

```
Block 0          Block 1          Block 2
[elem][elem]...  [elem][elem]...  [elem][elem]...
  ↑ 已消费完       ↑ 正在消费        ↑ 正在写入
  可回收复用        tail 在这里       head 在这里
```

- 每个 Block 是固定大小的数组，Block 之间用指针串起来
- 入队：往 head 所在 Block 写，Block 满了就分配新 Block 接上去
- 出队：从 tail 所在 Block 读，Block 内元素全部消费完后，整块放回空闲链表复用
- `{1000000}` 是预分配提示，不是硬上限——超了会继续分配新 Block，不会丢数据

构造函数里 queue 存的是 protobuf 深拷贝对象（带 string、动态内存），每个几百字节。100 万个槽位预分配约几百 MB。正常情况下 writer 持续消费，queue 不会积满。

#### 4.1.2 无锁并发机制：每个生产者独立写入链

5 个回调线程并发入队时的并发控制不依赖锁，而是**物理隔离**——每个生产者线程拥有独立的 Block 链：

```
回调线程 1 (ch 2011)    回调线程 2 (ch 2013)    回调线程 3 (ch 2015)
      │                       │                       │
      ▼                       ▼                       ▼
  [Block A1]──[Block A2]  [Block B1]──[Block B2]  [Block C1]
  线程1 自己的链          线程2 自己的链          线程3 自己的链
  ─────────────────────────────────────────────────────────
                    逻辑上是同一个 queue
```

**入队（生产者端）**：线程 1 只往自己的 Block A 链写，线程 2 只往自己的 Block B 链写，各自操作各自的内存，零竞争，无需 CAS 或锁。

**出队（消费者端）**：writer 线程调 `try_dequeue_bulk()` 时，轮询遍历所有生产者的链——取完一条链上当前可用的数据，再跳到下一条链。详见 4.2 节。

#### 4.1.3 生产者与消费者的并发控制

生产者写入和消费者读取可能同时操作同一条链上的同一个 Block。并发控制不靠锁，靠**原子变量做可见性控制**——写数据和公布数据分两步：

```
Block 内部:
  slot 0: [数据]  ✓ 已消费
  slot 1: [数据]  ✓ 已消费   ← dequeued = 2（原子变量，消费者维护）
  slot 2: [数据]  可读
  slot 3: [数据]  可读       ← committed = 4（原子变量，生产者维护）
  slot 4: [写入中]            ← 生产者正在写这里，尚未提交
  ...
```

- **生产者写入**：先把数据写进 slot（普通内存写入），再原子地将 committed +1（"公布"这条数据）
- **消费者读取**：读 committed 的值，只取 dequeued 到 committed 之间的 slot，然后原子地更新 dequeued

即使生产者正在写 slot 4，消费者只看到 committed = 4，只读 slot 0~3，不会读到写了一半的数据。

当 dequeued == committed == Block 容量时，整个 Block 消费完毕，回收到空闲链表供复用。

> 注：消费者只有一个 writer 线程，所以 dequeued 不存在多消费者竞争，原子操作只是为了保证对生产者的内存可见性（memory ordering），不需要 CAS 重试。

#### 4.1.4 内存回收

和环形数组不同（满了必须阻塞或丢弃），分块链表满了可以继续增长。消费完的 Block 不释放内存，而是放回空闲链表供后续入队复用，避免反复 malloc/free。

### 4.2 Writer 线程：串行取出 → 转换 → 写盘

#### 4.2.1 消费循环代码

```cpp
// insight_gateway.h:55-77
void process_queue() {
    while (running) {
        auto queue_size = queue.size_approx();        // ① 估算当前队列中有多少条
        if (queue_size > 0) {
            std::vector<T1> batch(queue_size);         // ② 分配 batch 容器
            size_t n = queue.try_dequeue_bulk(          // ③ 尽量取 queue_size 条
                batch.begin(), queue_size);
            for (const auto &stock_data : batch) {     // ④ 逐条处理
                T2 packed_data = stock_data.ParseFrom();
                data_writer->write_frame(&packed_data, sizeof(T2), ...);
            }
        }
        // queue_size == 0 时：不 sleep，直接再循环（busy-wait）
    }
}
```

#### 4.2.2 逐步解析

**① `size_approx()` — 估算队列大小**

返回的是一个近似值，不是精确值。因为调用的同时生产者还在并发入队，所以这个数字只是"大概有这么多"。返回 0 不代表真的空了（可能有数据正在提交），返回 1000 也不代表精确有 1000 条。这里用它只是为了给 batch 预分配一个大致够用的容器。

**② 分配 batch 容器**

`std::vector<T1> batch(queue_size)` — 创建一个能容纳 queue_size 个元素的 vector。注意这里 T1 是 protobuf 深拷贝对象（RawOrderData 等），每次循环都会 new/delete 这个 vector，有一定开销。

**③ `try_dequeue_bulk()` — 批量出队**

这是核心步骤。该函数的行为是：

```
遍历所有生产者的链:
  对于每条链:
    读取该链当前 Block 的 committed 值
    从 dequeued 位置开始，取到 committed 位置
    更新 dequeued
    如果整个 Block 取完了，移动到链上的下一个 Block
    如果已取够 max_count 条，提前返回
  移动到下一条链
返回实际取到的条数 n
```

具体举例，假设 3 个生产者线程的链状态如下：

```
线程1 的链:  Block[committed=3, dequeued=0]  →  Block[committed=1, dequeued=0]
             股票A, 股票B, 股票C                 股票G
线程2 的链:  Block[committed=2, dequeued=0]
             股票D, 股票E
线程3 的链:  Block[committed=1, dequeued=0]
             股票F
```

`try_dequeue_bulk(batch, 7)` 的执行过程：
1. 访问线程 1 的链：Block 1 取 A,B,C（dequeued 0→3，Block 消费完，回收），Block 2 取 G（dequeued 0→1），已取 4 条
2. 访问线程 2 的链：取 D,E（dequeued 0→2），已取 6 条
3. 访问线程 3 的链：取 F（dequeued 0→1），已取 7 条，达到 max_count，返回

最终 batch = `[A, B, C, G, D, E, F]`，返回 n = 7。

**关键特征**：同一条链上的数据在 batch 里保持连续和有序（A,B,C,G 都是线程 1 按入队顺序排列），不同链之间按遍历顺序拼接。这就是跨 channel 乱序的直接成因——batch 里的顺序和数据的实际到达时间无关。

**④ 逐条处理**

writer 线程按 batch 顺序依次执行 `ParseFrom()` + `write_frame()`，写入 journal。每次 `write_frame()` 调用 `getNano()` 获取墙上时间作为帧时间戳，单线程串行写入保证 nano 严格递增。

#### 4.2.3 `size_approx()` 不准会怎样？

`size_approx()` 是近似值，理论上可能偏大或偏小。

**估小了**（比如估了 500，实际有 800）：这次只取 500 条，剩下的下次循环再取。不会丢数据。

**估大了**（比如估了 1000，实际只有 800）：`try_dequeue_bulk()` 只取到 800 条，返回 `num_dequeued = 800`。但 batch vector 有 1000 个元素，位置 800~999 是默认构造的空对象。

当前代码直接遍历了整个 batch：

```cpp
for (const auto &stock_data : batch) {     // 遍历全部 1000 个，而不是前 800 个
    T2 packed_data = stock_data.ParseFrom();  // 空对象也被 ParseFrom
```

正确写法应该用返回值 `num_dequeued` 控制遍历范围：

```cpp
for (size_t i = 0; i < num_dequeued; ++i) {
    T2 packed_data = batch[i].ParseFrom();
```

如果空对象被 `ParseFrom()` 处理，会走到 `StockOrderData` 构造函数（`insight_types.h:741`），默认构造的 protobuf 没有 `mdorder` 字段，直接 `throw std::invalid_argument`。writer 线程内没有 try-catch，未捕获异常导致 `std::terminate()`，进程崩溃。

**但实际上这个问题不会触发。** 原因是 `size_approx()` 在这个场景下实际只会偏小、不会偏大：

- 生产者 enqueue 的流程是：先写数据到 Block，再原子递增计数器
- 消费者读 `size_approx()` 时，可能有数据已写入但计数器还没递增——看到的比实际少
- 消费者自己的 dequeue 递减是同线程操作，对自己立即可见，不会滞后

所以 `size_approx()` 返回 1000 时，queue 里实际至少有 1000 条（甚至更多）。`try_dequeue_bulk()` 总能取满，`num_dequeued == queue_size`，默认构造的空对象不会被遍历到。

**结论**：代码写法不严谨（应该用 `num_dequeued` 来遍历），但在实际运行中不会触发问题。线上程序收盘后仍在运行且从未因此崩溃，也验证了这一点。属于代码风格问题，不是真正的 bug。

每种数据类型（tick/order/trade）有独立的 writer 线程和 JournalWriter 实例。write_frame 内部调 `getNano()` 获取当前时间作为帧时间戳，单线程串行写入保证 nano 严格递增。

### 4.3 nano_timestamp 的双重语义

| 场景 | nano_timestamp 来源 | 含义 |
|------|-------------------|------|
| 线上实盘 | `frame->getNano()` | writer 线程写入 journal 的墙上时间 |
| 本地 replay | `frame->getExtraNano()` | 原始 journal 中保存的历史写入时间 |

在 `realtime_parquet` 中统一为：
```cpp
int64_t nano = (extra_nano > 0) ? extra_nano : frame->getNano();
```

## 5. 架构分析与可优化点

### 5.1 当前架构的并行度错配

| 阶段 | 线程数 | 工作量 |
|------|--------|--------|
| 回调：protobuf 深拷贝 + 入队 | **5 线程并行** | 轻（内存拷贝） |
| Writer：ParseFrom + write_frame | **1 线程串行** | 重（stoi、类型转换、mmap 写入） |

CPU 密集的 ParseFrom 在单线程执行，反而是较轻的拷贝 + 入队被 5 线程并行化。并行加在了轻的一端。

### 5.2 可选优化方案

**方案 A：回调中直接 ParseFrom，queue 存 packed struct**

```
5 回调线程 → ParseFrom（并行）→ enqueue(packed struct) → 1 writer → write_frame
```

好处：
- ParseFrom 被 5 线程并行化
- 省掉 protobuf 深拷贝（直接从 SDK 给的对象上 ParseFrom）
- queue 存固定大小的小 struct（几十字节），而非 protobuf 大对象
- 回调可能反而更快（ParseFrom 是纯栈操作，零内存分配；protobuf 深拷贝需要堆分配）

**方案 B：回调单线程 + 直接写 journal**

```
1 回调线程 → ParseFrom → write_frame → Journal
```

好处：最简单，完全保序，无 queue 无额外线程。
代价：回调占用时间变长，高峰期可能导致 SDK 内部积压。

## 6. realtime_parquet 的应对策略

基于以上分析，`realtime_parquet` 用 **nano_timestamp 的分钟**而非 exchange_time 的分钟来切分文件：

```cpp
// main.cpp:203
int minute = nano_to_hhmm(nano);  // 从 nano 提取 HHMM
if (minute != ch.current_minute) {
    rotate_channel(ch, minute, nano);
}
```

- nano 严格递增 → 分钟只会前进，不会回退 → 无需 forward-only 保护
- 文件名反映数据到达时间（如 `order_0925.parquet` = nano 在 09:25 这分钟到达的数据）
- 文件内容可能包含不同 exchange_time 的数据（如 SH 集合竞价 order 的 exchTime 横跨 09:15~09:24，但全部在 09:25 到达）
