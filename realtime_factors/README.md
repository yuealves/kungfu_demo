# Realtime Factors 实时因子计算框架

## 1. 框架简介
`realtime_factors` 是一个基于 KungFu Journal 的极低延迟实时因子计算框架。它通过“进程内增量同步 + 共享内存快照池”架构，实现了在主进程 **绝对零抖动** 的前提下，为下游提供物理时间对齐的全量因子快照。

---

## 2. 系统中的四个关键角色

### 角色 1：时序引擎 (TS-Engine) —— 极速更新主线程
*   **职责**：负责逐笔行情处理（Order/Trade），计算并更新基础时序因子。
*   **机制**：计算结果以增量（Delta）形式写入私有的 **内存环形缓冲区 (Internal Delta Buffer)**。
*   **性能**：主循环仅执行纳秒级原子操作，单次更新开销 < 20ns。

### 角色 2：快照编排器 (Snapshot-Composer) —— 特征一致性核心
*   **职责**：
    1.  **日志重放**：回放增量日志，维护私有的全量 Checkpoint 镜像。
    2.  **截面合成**：计算衍生截面因子，定格全量因子快照。
    3.  **发布快照**：将快照发布至共享内存的 **Snapshot Region**。
*   **发布机制 (Sequence-based Search)**：
    *   **确定性映射**：物理槽位由 `LatestSeq % N` 唯一确定。
    *   **非阻塞跳转**：若 `LatestSeq % N` 对应的槽位被读者引用（Pin），则**持续自增序号**跳过该槽位，确保写者永不阻塞。
    *   **流量保证**：由于下游 Reader 数量有限（通常 < 10，可通过注册机制强制要求 Reader 数目小于槽位数 $N$），Snapshot-Composer 永远能找到空闲槽位进行写入，实现了**逻辑上的绝对无锁写**。

### 角色 3：下游策略进程 (Consumer) —— 零拷贝原地计算者
*   **职责**：执行交易逻辑与模型推理。
*   **机制：槽位锁定 + 零拷贝 (Zero-Copy)**：
    1.  **原子引用 (Pin)**：获取最新快照序号并增加其引用计数（Pin）。
    2.  **原地计算**：直接在共享内存上映射 NumPy 数组进行模型推理，**无需执行数据拷贝**。
    3.  **计算释放**：推理完成后减少引用计数（Unpin），释放槽位供写者回收。

### 角色 4：持久化器 (Persister) —— 后台归档线程
*   **职责**：将实时因子快照定期落盘。
*   **机制：异步写盘**：
    1.  采用与角色 3 相同的“槽位锁定”机制。
    2.  为隔离 I/O 耗时，先执行 `memcpy` 至私有 Buffer 后立即解锁，随后在后台独立线程执行 Parquet 写入。

## 3. 核心设计深度解析

### 3.1 为什么放弃 fork()？
*   **页表开销**：在 10GB+ 的大内存环境下，`fork()` 拷贝页表会造成数百微秒的确定性抖动。
*   **全线程优势**：通过“序号跳转 + 槽位引用计数”实现了比 `fork()` 更轻量、新鲜度更高的一致性快照方案。

### 3.2 绝对的一致性 (Atomicity)
*   **消除特征撕裂**：每个 Slot 存储的是同一时刻的全量切片。`Snapshot-Composer` 确保了原始值与衍生值在逻辑序号上完全对齐。

### 3.3 NumPy 友好的 SHM 布局
*   **实时预览区 (Live TS Region)**：行优先 (Row-Major)，支持 Seqlock，满足单股票极速监控需求。
*   **快照发布区 (Snapshot Region)**：列优先 (Column-Major)。支持 Python **零拷贝 (Zero-Copy)** 内存映射，极大提升向量化运算性能。

## 4. 架构权衡：无抖动与稳定性

| 风险点 | 应对方案 | 预期延迟 |
| :--- | :--- | :--- |
| **主进程 Jitter** | 内部增量日志 (Atomic Append) | < 20ns |
| **读写冲突** | 槽位引用计数 (Pinning) | < 100ns |
| **槽位死锁/套圈** | **序号自增跳转 (LatestSeq++)** 机制 | 零影响 |
| **读取开销** | **共享内存零拷贝 (Zero-Copy)** | 0μs |
