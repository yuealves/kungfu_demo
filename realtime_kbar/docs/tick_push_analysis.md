# Tick 推送特征分析（2026-03-17 实盘数据）

## 背景

kbar_realtime 需要确定何时 flush 一分钟的 kbar CSV。核心问题：**分钟边界后，什么时候能确认所有股票的数据已到齐？**

为此分析了华泰 Insight L2 tick 快照的推送特征。

数据源：`/data/tickl2_data_ht2_raw/2026-03-17/` （10 个 tick parquet 文件，全天数据）

---

| 交易所 | 推送频率 | 每秒股票数 | 全天唯一股票数 |

| 深圳 (SZ) | 每 3 秒一批 | ~2900（一次性全推） | 2922 |

## 4. 分钟边界附近的时间线

以 09:39→09:40 边界为例（HHMM 从 939 变到 940）：

```
09:39:57.000  SZ ~2900 + SH ~640    ← 最后一个 HHMM=939 的 SZ 批次
09:39:58.000  SH ~640               ← SH 继续推
09:39:59.000  SH ~640               ← SH 本轮最后一批（至此 SH 2307 只全部到齐）
09:40:00.000  SZ ~2900 + SH ~640    ← HHMM 变为 940，SZ 新一轮开始
09:40:01.000  SH ~640               ← SH 新一轮
09:40:02.000  SH ~640               ← SH 新一轮（至此 SH 新一轮 2307 只全部到齐）
09:40:03.000  SZ ~2900 + SH ~640    ← SZ 第二轮
```

**关键时序**：
- SZ 在 09:40:00 一次性推完 ~2900 只（bar 940 的 SZ 部分瞬间到齐）
- SH 在 09:40:00 + 09:40:01 + 09:40:02 分 3 批推完 ~2307 只（bar 940 的 SH 部分需要 ~2 秒）
- 因此 bar 940 的全部 ~5270 只股票在 **09:40:02** 到齐，延迟约 **2 秒**

## 6. 当前 kbar_realtime flush 方案

### 流式 flush（秒级 exchange_time 变化触发）

```
HHMM 跨分钟时（如 093957→094000）：记下 pending_flush_bar = 940
下一个 HHMMSS 变化时（094000→094001）：执行 flush bar 940
```

延迟 ~1 秒（到下一个 SH 批次）。但此时只有 094000 那秒的 SH 股票（~640 只）完成了 bar 切换，其余 ~1660 只 SH 股票还没收到 HHMM=940 的 tick。

**实测结果**（replay 回放）：每个 bar 5270 stocks，但这是因为 replay 数据只覆盖半天，所有 bar 实际由 finalize 输出（idle timeout 触发），不是流式 flush。流式 flush 的正确性未充分验证。

### Finalize（EOD 退出）

- 收到 exchange_time > 150500000 时触发
- 或 60 秒无数据 idle timeout 触发
- 使用 `finalize_all_current_only()`：只 emit 当前 in-progress bar，不填充空分钟
- 关闭 callback → finalize → 收集所有 bar → flush_all 一次性写出

## 7. 待解决：流式 flush 的触发时机

### 问题

SH 分 3 秒推完一轮，分钟边界后需要等 ~2 秒才能确认所有 SH 股票到齐。当前秒级触发方案延迟 ~1 秒，可能导致部分 SH 股票缺失。

### 讨论过的方案

**方案 A：秒级 exchange_time 变化触发**
- 延迟 ~1 秒，可能丢 ~1600 只 SH 股票
- 已实现，需要验证完整数据下的正确性

**方案 B：凑够 SZ + SH 股票数**
- 开盘时学习 SZ 和 SH 各自的预期数量
- 分钟边界后分别计数，两边都齐了就 flush
- SH 完全稳定（2307），可直接用作阈值
- SZ 偶尔缺少量 B 股（2911~2922），需设略低阈值或容忍
- 延迟取决于最慢的一边到齐的时间（~2 秒）

**方案 C：只用 SH 计数触发**
- SZ 一定比 SH 先到齐（SZ 在 3 秒整点一次性全推）
- 只需等 SH 的 2307 只到齐（~2 秒）
- 但需要确认 SZ 确实总是先于或同时于 SH 到齐

### 方案 C 详细设计与验证（2026-03-23）

#### 方案描述

HHMM 跨分钟后，只数新分钟内 SH 的唯一股票数。当 SH 凑齐 `expected_sh_count` 时，说明该分钟全部有效数据已到齐，立即 flush。

原理：SH 每秒推 ~640 只，3 秒推完一轮 2307 只，是最后到齐的一方。SZ 在 3 秒整点一次性推完 ~2900 只，必然先于 SH 到齐。所以 SH 凑齐 = 全市场凑齐。

`expected_sh_count` 不硬编码，每天开盘时从第一个完整分钟动态学习（新股上市/退市会导致数字变化）。

#### 前提条件与验证

**前提 1：SH 每分钟唯一股票数恒定**

即同一天内，每个交易分钟推送的 SH 唯一股票数完全一致，可以用第一个分钟的值作为 threshold。

验证（10 个交易日，2026-03-09 至 2026-03-20）：

```
2026-03-09: 242 分钟, SH per_min=[2307,2307] stable=✓
2026-03-10: 242 分钟, SH per_min=[2307,2307] stable=✓
2026-03-11: 242 分钟, SH per_min=[2307,2307] stable=✓
2026-03-12: 242 分钟, SH per_min=[2307,2307] stable=✓
2026-03-13: 242 分钟, SH per_min=[2307,2307] stable=✓
2026-03-16: 242 分钟, SH per_min=[2307,2307] stable=✓
2026-03-17: 242 分钟, SH per_min=[2307,2307] stable=✓
2026-03-18: 242 分钟, SH per_min=[2307,2307] stable=✓
2026-03-19: 242 分钟, SH per_min=[2307,2307] stable=✓
2026-03-20: 242 分钟, SH per_min=[2307,2307] stable=✓
```

结论：**10 天 × 242 分钟 = 2420 个样本，全部 2307，零波动。✓**

**前提 2：每分钟内 SZ 总是先于 SH 完成推送**

即 SZ 该分钟最后一条 tick 的 exchange_time ≤ SH 该分钟最后一条 tick 的 exchange_time。这保证了 SH 到齐时 SZ 已经到齐。

验证（10 个交易日）：

```
2026-03-09 ~ 2026-03-20: 全部 10 天 × 242 分钟，SZ_last <= SH_last ✓
```

典型时差：SH 比 SZ 晚 2 秒（SZ 在 xx:yy:57 推完，SH 在 xx:yy:59 推完）。

结论：**2420 个样本无一例外，SZ 总是先于 SH 到齐。✓**

**前提 3：SH 的每日股票总数会变化，不能硬编码**

验证：10 天内 SH 恒为 2307，SZ 恒为 2922，但这只是因为这 10 天没有新股上市/退市。长期来看数字会变化。

```
2026-03-09 ~ 2026-03-20: SH=2307, SZ=2922 (10 天一致)
```

结论：**不能硬编码 2307。每天开盘时从第一个完整交易分钟动态学习。**

**前提 2 的 nano_timestamp 验证（关键！）**

用 exchange_time 比较 SZ/SH 到齐顺序是**不够的**。必须用 nano_timestamp（journal 实际写入顺序）验证。

结果（10 天）：按 nano_timestamp 顺序，SH 凑齐 2307 时，SZ 有 ~12 只股票尚未首次出现在该分钟。

```
2026-03-09 ~ 2026-03-20: 237/242 分钟 SZ 有少量股票晚于 SH 到齐
典型: 12 只 SZ 股票延迟 1~22 秒（包括 B 股和少量 A 股）
```

示例（HHMM=940）：SH 在 09:40:02 凑齐。以下 SZ 股票在之后才到：
```
Symbol=200505  Time=94030000  delay=1412ms
Symbol=  2231  Time=94036000  delay=6719ms
Symbol=300344  Time=94036000  delay=7599ms
Symbol=200011  Time=94051000  delay=22454ms
```

**结论：这不影响方案 C 的正确性。** 这些股票在 SH 凑齐时还没收到该分钟的 tick，bar 尚未 emit，不可能被 flush 漏掉。它们的 bar 会在后续 tick 到达时 emit，被下一分钟的 flush 捕获。数据准确性不受影响，只是这 ~12 只股票的 bar 延迟一个 flush 周期。

**关于少量 SZ B 股偶尔缺席**

SZ 每分钟有 0~11 只 B 股（200xxx）不被推送。这不影响方案 C：这些股票该分钟没有新 tick，不会产生新 bar，等不等都一样。SH 凑齐时该 flush 的数据全部齐了。

#### 实现要点

1. 判断 SH/SZ：`Symbol >= 400000` 为 SH，`< 400000` 为 SZ
2. 学习 `expected_sh_count`：第一个 HHMM >= 931 的完整分钟中 SH 唯一股票数
3. 分钟边界检测：当 `compute_bar_minute(Time)` 变大时，进入新分钟
4. 计数触发：HHMM 跨分钟后，用 set/counter 记录新分钟内 SH 唯一 Symbol，凑齐后 flush 上一分钟的 bar
5. 兜底：收盘后 idle timeout 或 eod 触发 finalize

#### Flush 延迟

SH 一轮 2307 只分 3 秒推完（每秒 ~640 只）。HHMM 跨分钟后，SH 新一轮从 xx:00 开始，到 xx:02 推完。延迟 **~2 秒**。
